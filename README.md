# Отчёт по проекту «Система учёта заявок сотрудников»

## 1. Краткое описание реализованного бизнес-процесса

Система реализует учёт заявок сотрудников с жизненным циклом, состоящим из трёх последовательных статусов:

```
Новая → В работе → Выполнена
```

Каждая заявка создаётся сотрудником-автором, может быть назначена на сотрудника-исполнителя, содержит описание и срок выполнения (дедлайн). Работа с заявкой строится вокруг четырёх операций:

1. **Создание заявки** — фиксируются автор, описание, срок выполнения и (опционально) исполнитель. Заявке присваивается уникальный номер (например, `T-1042`) и статус «Новая».
2. **Назначение/переназначение исполнителя** — можно выполнить на любом этапе жизненного цикла, кроме статуса «Выполнена».
3. **Перевод в статус «В работе»** — возможен только из статуса «Новая».
4. **Завершение заявки** — возможен только из статуса «В работе».

Параллельно с управлением отдельными заявками система поддерживает:

- **Справочник сотрудников** (ФИО, подразделение, должность) с возможностью создания и редактирования (смена подразделения/должности).
- **Просмотр списка заявок с фильтрацией** по статусу, исполнителю, подразделению исполнителя и признаку просроченности (`Deadline < текущее время`).
- **Аналитическую отчётность**: количество заявок по каждому статусу, количество просроченных заявок, количество выполненных заявок в разрезе исполнителей.

Взаимодействие с системой происходит через интерактивное консольное меню, обращающееся исключительно к сервисам Application-слоя — сама консоль не содержит бизнес-логики и не обращается к базе данных напрямую.

## 2. Описание реализованных бизнес-правил

### 2.1 Правило допустимых переходов статуса (State pattern)

Жизненный цикл заявки реализован через паттерн **State**: каждому статусу соответствует отдельный класс (`NewState`, `InProgressState`, `CompletedState`), реализующий интерфейс `ITicketState`. Именно конкретный класс состояния определяет, какие переходы допустимы:

| Текущий статус | `TryStartWork()` | `TryComplete()` | Назначение исполнителя |
|---|---|---|---|
| Новая (`NewState`) | ✅ → В работе | ❌ (нельзя завершить напрямую) | ✅ разрешено |
| В работе (`InProgressState`) | ❌ (уже не новая) | ✅ → Выполнена | ✅ разрешено |
| Выполнена (`CompletedState`) | ❌ | ❌ | ❌ запрещено |

Ключевое бизнес-правило из требований — **заявку нельзя перевести из «Новая» напрямую в «Выполнена»** — гарантируется на уровне домена: у `NewState` метод `TryComplete()` безусловно возвращает `false`, минуя статус «В работе» перейти в «Выполнена» невозможно ни при каких условиях.

Если операция недопустима в текущем статусе, Application-слой перехватывает `false` от домена и выбрасывает `InvalidTicketStateTransitionException` — отдельный тип исключения, который консольный UI перехватывает и показывает пользователю понятное сообщение, не прерывая работу программы.

### 2.2 Правило запрета переназначения исполнителя завершённой заявки

Метод `TryAssignExecutor` также делегируется текущему состоянию: `CompletedState.TryAssignExecutor` всегда возвращает `false`. Это защищает от изменения исполнителя у уже закрытой заявки — попытка такой операции завершится тем же `InvalidTicketStateTransitionException`.

### 2.3 Правило валидации сущностей на уровне конструктора

И `Employee`, и `Ticket` не позволяют создать себя в невалидном состоянии:

- ФИО, подразделение и должность сотрудника не могут быть пустыми (`ArgumentException` в конструкторе `Employee`).
- Описание заявки не может быть пустым.
- Срок выполнения (`Deadline`) должен быть в будущем на момент создания заявки.

Эти проверки происходят в момент конструирования объекта, поэтому невозможна ситуация, при которой в системе появляется заявка или сотрудник с некорректными данными — независимо от того, из какого сценария использования (консоль, будущий веб-API) идёт вызов.

### 2.4 Правило ссылочной целостности «Автор/Исполнитель — существующий сотрудник»

При создании заявки и при назначении исполнителя Application-слой (`TicketCreatorService`, `TicketAssignerService`) явно проверяет существование сотрудника по `Id` через `IEmployeeRepository.GetById`, и при отсутствии выбрасывает `EntityNotFoundException`. Это предотвращает создание заявок со ссылкой на несуществующего сотрудника ещё до попытки записи в БД (хотя на уровне БД это дополнительно подкреплено внешними ключами `AuthorId`/`ExecutorId → Employees.Id`).

### 2.5 Правило определения просроченности

Заявка считается просроченной, если текущее время больше её `Deadline` (`Ticket.IsOverdue(DateTime currentUtcTime)`). Расчёт статуса просроченности не хранится как отдельное поле в БД, а вычисляется динамически — это исключает риск рассинхронизации данных (например, если бы флаг «просрочена» хранился отдельно и не обновлялся).

## 3. Описание выполненной оптимизации базы данных и результатов замеров производительности

### 3.1 Целевой запрос

Согласно требованиям, замерялся запрос: «вывести все просроченные заявки конкретного исполнителя, находящиеся в статусе «В работе», отсортированные по сроку выполнения»:

```sql
SELECT *
FROM Tickets
WHERE ExecutorId = :executorId
  AND Status = 1        -- InProgress
  AND Deadline < NOW()
ORDER BY Deadline;
```

Объём тестовых данных: **1 000 000 заявок**, **1000 сотрудников**.

### 3.2 Замер «до» оптимизации

Без дополнительных индексов (кроме `PRIMARY KEY` и `FOREIGN KEY`) Postgres вынужден выполнять **последовательное сканирование (`Seq Scan`)** всей таблицы `Tickets`, проверяя условие `WHERE` построчно для всех 1 000 000 записей, и только затем сортировать отфильтрованный результат в памяти/на диске (`Sort`).

```
Sort  (cost=... rows=~333 width=...)
  Sort Key: deadline
  ->  Seq Scan on tickets  (cost=0.00..XXXXX.XX rows=~333 width=...)
        Filter: (executorid = :executorId AND status = 1 AND deadline < now())
        Rows Removed by Filter: ~999667
Planning Time: X.XXX ms
Execution Time: <ЗАМЕРЕННОЕ_ВРЕМЯ_ДО> ms
```

> Точные числа `Execution Time` и `cost` зависят от железа и текущей загрузки БД — их нужно подставить из фактического вывода `EXPLAIN ANALYZE`, полученного на своей машине (см. п.7 инструкции по запуску). На объёме в 1 млн строк без индекса типичное время выполнения такого запроса на локальной машине обычно составляет от нескольких десятков до нескольких сотен миллисекунд, в зависимости от диска и объёма оперативной памяти под кэш.

### 3.3 Внесённые изменения (оптимизация)

Создан составной индекс:

```sql
CREATE INDEX IX_Tickets_Executor_Status_Deadline
    ON Tickets (ExecutorId, Status, Deadline);
```

**Почему выбран именно такой порядок колонок:**

1. `ExecutorId` и `Status` участвуют в запросе как условия **точного равенства** (`=`) — они стоят первыми, чтобы индекс мог сразу сузить диапазон поиска до узкого поддерева B-tree.
2. `Deadline` используется одновременно и в условии диапазона (`<`), и в `ORDER BY` — будучи последней колонкой составного индекса, она позволяет Postgres не только быстро найти нужные строки, но и **отдать их уже в отсортированном порядке**, не выполняя отдельную операцию `Sort`.

Дополнительно был создан индекс `IX_Employees_Department` — он не влияет на этот конкретный запрос (в нём фильтрации по подразделению нет), но ускоряет другой востребованный сценарий из требований: фильтрацию списка заявок по подразделению исполнителя (`GetFiltered(department: ...)`), где идёт `JOIN` с `Employees`.

### 3.4 Замер «после» оптимизации

С индексом Postgres использует **`Index Scan`** (или `Index Only Scan`, если бы запрашивались только колонки, входящие в индекс) вместо полного скана таблицы: сразу переходит к нужному поддиапазону B-tree по `(ExecutorId, Status)`, а условие `Deadline < NOW()` и сортировка обрабатываются в рамках уже упорядоченного индекса — без отдельного шага `Sort`.

```
Index Scan using ix_tickets_executor_status_deadline on tickets
  (cost=0.XX..XXX.XX rows=~333 width=...)
  Index Cond: (executorid = :executorId AND status = 1 AND deadline < now())
Planning Time: X.XXX ms
Execution Time: <ЗАМЕРЕННОЕ_ВРЕМЯ_ПОСЛЕ> ms
```

### 3.5 Итоговое сравнение

| Метрика | До оптимизации | После оптимизации |
|---|---|---|
| Метод доступа | Seq Scan + Sort | Index Scan (сортировка не требуется) |
| Просмотрено строк | ~1 000 000 (вся таблица) | только строки, удовлетворяющие `ExecutorId + Status` |
| Execution Time | `<ПОДСТАВИТЬ ИЗ ЗАМЕРА>` мс | `<ПОДСТАВИТЬ ИЗ ЗАМЕРА>` мс |

**Почему это ускоряет выполнение:** без индекса стоимость запроса линейно растёт с размером таблицы — независимо от того, сколько строк реально удовлетворяет фильтру, Postgres обязан проверить каждую строку таблицы. С составным индексом стоимость запроса зависит не от размера всей таблицы, а от размера **результата** (количества заявок конкретного исполнителя в нужном статусе) и глубины B-tree (логарифмическая величина от размера индекса) — при равномерном распределении заявок между 1000 исполнителей это на несколько порядков меньше полного скана таблицы. Дополнительный выигрыш даёт отсутствие отдельного шага сортировки, поскольку строки уже физически упорядочены по `Deadline` внутри найденного диапазона индекса.

### 3.6 Дополнительные решения по структуре БД (кратко)

- `AuthorId`/`ExecutorId` в `Tickets` — внешние ключи на `Employees`, а не дублирование ФИО/подразделения в самой заявке: устраняет избыточность и соответствует 3NF (изменение данных сотрудника не требует обновления исторических заявок).
- `Department`/`Position` хранятся как атрибуты `Employees`, а не в отдельных справочных таблицах: в рамках требований задачи это неизменяемые в структуре БД строковые характеристики без собственного жизненного цикла и связей — вынесение в отдельные таблицы увеличило бы сложность схемы без практической пользы для заявленной функциональности.
- `Status` хранится как `SMALLINT` с `CHECK`-ограничением, а не отдельной таблицей-справочником статусов: набор статусов фиксирован и определяется доменной моделью (State pattern) на уровне приложения, а не данными в БД.
- Номер заявки (`Number`) генерируется через `SEQUENCE` (`ticket_number_seq`) — атомарная генерация на уровне СУБД исключает гонки при конкурентном создании заявок без явных блокировок таблицы.


# Инструкция по запуску и проверке проекта

## 1. Предварительные требования

- Docker + Docker Compose
- .NET SDK 8.0+ (`dotnet --version`)
- (опционально) `psql` клиент или любой GUI (DBeaver/pgAdmin) для ручных проверок и `EXPLAIN ANALYZE`

## 2. Структура проекта

```
/
├── docker-compose.yml
├── db/
│   └── migrations/
│       ├── 001_schema.sql
│       ├── 002_seed_employees.sql
│       └── 003_seed_tickets.sql
└── src/
    ├── TicketSystem.Core/
    ├── TicketSystem.Application/
    ├── TicketSystem.Infrastructure/
    └── TicketSystem.ConsoleApp/
```

## 3. Поднятие базы данных

```bash
docker compose up -d
```

Дождаться готовности контейнера:

```bash
docker compose ps
```

В колонке `STATUS` должно появиться `healthy` (это займёт ~10-20 секунд — за это время Postgres успевает инициализироваться и выполнить все `.sql`-файлы из `db/migrations`, включая генерацию 1000 сотрудников и 1 000 000 заявок).

Первая инициализация с генерацией 1 млн заявок может занять от нескольких секунд до пары минут — это нормально, следить за прогрессом можно через:

```bash
docker compose logs -f postgres
```

## 4. Проверка загрущки данных

```bash
docker exec -it ticketsystem-postgres psql -U postgres -d ticketsystem -c "SELECT count(*) FROM Employees;"
docker exec -it ticketsystem-postgres psql -U postgres -d ticketsystem -c "SELECT count(*) FROM Tickets;"
```

Ожидаемый результат: `1000` и `1000000` соответственно.

## 5. Сборка и запуск консольного приложения

Из корня проекта:

```bash
cd src/TicketSystem.ConsoleApp
dotnet restore
dotnet build
```

Задать строку подключения (по умолчанию в `Program.cs` уже указана строка, совпадающая с `docker-compose.yml`, но лучше явно):

**Linux/macOS:**
```bash
export TICKETSYSTEM_CONNECTION="Host=localhost;Database=ticketsystem;Username=postgres;Password=postgres"
```

**Windows (PowerShell):**
```powershell
$env:TICKETSYSTEM_CONNECTION="Host=localhost;Database=ticketsystem;Username=postgres;Password=postgres"
```

Запуск:

```bash
dotnet run
```

Должно появиться интерактивное меню:

```
=== Система учёта заявок сотрудников ===
1. Добавить сотрудника
2. Список сотрудников
3. Создать заявку
4. Назначить исполнителя
5. Взять заявку в работу
6. Завершить заявку
7. Список заявок (с фильтрами)
8. Отчёт
0. Выход
>
```

## 6. Проверка функциональности (сценарий)

Рекомендуемый порядок ручной проверки, покрывающий все требования docs.txt:

1. **Пункт 1** — создать сотрудника (запомнить выведенный `Id`, понадобится далее).
2. **Пункт 2** — убедиться, что созданный сотрудник появился в списке.
3. **Пункт 3** — создать заявку, указав `Id` автора из шага 1. Оставить исполнителя пустым (Enter). Запомнить `Id` заявки из вывода.
4. **Пункт 7** — вывести список заявок без фильтров, убедиться, что новая заявка в статусе "Новая".
5. **Пункт 4** — назначить исполнителя на заявку (можно того же сотрудника или любой существующий `Id` из seed-данных — их можно достать через `SELECT Id FROM Employees LIMIT 5;` в psql).
6. **Пункт 6** — попытаться сразу завершить свежесозданную заявку (минуя "В работе") — ожидается **ошибка бизнес-правила** (`InvalidTicketStateTransitionException`): заявку нельзя перевести в "Выполнена" из "Новая". Это ключевая проверка правила из docs.txt.
7. **Пункт 5** — перевести заявку в "В работе" — должно пройти успешно.
8. **Пункт 6** — повторно завершить заявку — теперь должно пройти успешно.
9. **Пункт 7** — проверить фильтры по очереди: по статусу (`2` — Выполнена), по `executorId`, по `department`, по флагу "только просроченные" (`y`) — используя seed-данные, где просроченные заявки заведомо есть.
10. **Пункт 8** — вывести отчёт, убедиться, что:
    - сумма `StatusCounts` по всем статусам совпадает с `1 000 001` (1 млн seed + созданная вручную заявка);
    - `OverdueCount` больше нуля;
    - `ExecutorCompletedCounts` содержит хотя бы вашего исполнителя с `CompletedCount = 1`.

### 7. Замер производительности целевого запроса

Подключиться к БД напрямую:

```bash
docker exec -it ticketsystem-postgres psql -U postgres -d ticketsystem
```

Получить `Id` реального исполнителя, у которого есть заявки в статусе "В работе":

```sql
SELECT ExecutorId, count(*) 
FROM Tickets 
WHERE Status = 1 AND ExecutorId IS NOT NULL 
GROUP BY ExecutorId 
ORDER BY count(*) DESC 
LIMIT 1;
```

**Замер "до" оптимизации** — временно удалить индекс:

```sql
DROP INDEX IF EXISTS IX_Tickets_Executor_Status_Deadline;

EXPLAIN ANALYZE
SELECT *
FROM Tickets
WHERE ExecutorId = '<подставить-guid-из-предыдущего-запроса>'
  AND Status = 1
  AND Deadline < NOW()
ORDER BY Deadline;
```

Сохранить вывод (обратить внимание на `Seq Scan`, `Execution Time`).

**Замер "после" оптимизации** — вернуть индекс и повторить тот же запрос:

```sql
CREATE INDEX IX_Tickets_Executor_Status_Deadline
    ON Tickets (ExecutorId, Status, Deadline);

EXPLAIN ANALYZE
SELECT *
FROM Tickets
WHERE ExecutorId = '<тот-же-guid>'
  AND Status = 1
  AND Deadline < NOW()
ORDER BY Deadline;
```

Сохранить вывод (ожидается `Index Scan` вместо `Seq Scan`, значительно меньший `Execution Time`).

Оба вывода `EXPLAIN ANALYZE` (до/после) вставить в отчёт как есть — это и есть требуемое docs.txt "описание оптимизации и результатов замеров".
